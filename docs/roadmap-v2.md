## 1. 정의와 원칙

### 한 줄 정의

한 대학 캠퍼스의 길고양이를 지도 기반으로 아카이빙하는 서비스.
"저 고양이 이름 뭐지?"라는 호기심으로 들어와서, "3시간 전 밥 먹음"을 보고 돌봄으로 이어지게 만든다.

### 차별점

급식 로그를 **개인 관리 기록이 아니라 낯선 사람에게 보내는 공개 신호**로 다룬다. 그리고 고정 급식소를 만들지 않는다.

### 원칙 (기능 판단 기준)

1. **"찾아가게" 만들지 말고 "우연히 만났을 때 알아보게" 만든다.**
   - **금지** — 실시간 위치, 위치 기반 푸시/알림, 경로 안내, 실시간 거리 갱신, 특정 개체 추적
   - **허용** — 1시간 지연된 목격 기록을 조회·정렬·필터하는 정적 화면
   - 주변 고양이 보기(§2.7-5)는 허용 쪽이다. **지연을 우회하는 순간 금지 쪽으로 넘어간다**
2. **급식소는 만들지 않는다.** 고정 급식 장소 등록 기능 없음.
3. **급식 기록은 장소가 아니라 고양이 개체에 붙는다.**
4. **사진의 EXIF(GPS 포함)는 무조건 제거한다.**
5. **목격 좌표는 저장 시점에 50m 격자로 반올림한다.** 정밀 좌표는 DB에 존재하지 않는다.
6. **로그인만 하면 모두 고양이 등록할 수 있다.**

> 원칙 1은 2026-07-21에 문구를 좁혔다. 원래 "실시간 위치 알림·경로 안내 없음"이 §3-5(주변 고양이 보기)와 정면 충돌했다. 금지 대상은 **"지금 여기 있다"**이지 **"이 근처에 이런 애들이 산다"**가 아니다.

> 원칙 5는 v1에서 "50~100m 퍼징"이라고만 적혀 있어서 구현이 두 번 뒤집혔다. **랜덤 오프셋이 아니라 결정적 라운딩이다.**

## 2. 확정 사항 — 다시 논의하지 않는다

### 2.1 인증 · 회원가입 (2026-07-19 확정)

**두 갈래를 모두 제공한다.**

| 방식 | 회원가입 | 이후 로그인 |
|---|---|---|
| **이메일** | 이메일 OTP 인증 **1회** → 비밀번호 설정 | 이메일 + 비밀번호 |
| **카카오** | 카카오 인증 (`account_email` 동의항목 포함, §2.8-13) | 카카오 인증 (계속) |

**두 방식 공통:**

- 가입 시 앱 자체 프로필을 새로 만든다
- **집사 이름(닉네임)은 무작위로 자동 생성**한다
- 카카오의 닉네임·프로필 사진은 **저장하지도, 화면에 그리지도 않는다.** 카카오는 오직 인증 통과용
- **대학 이메일 도메인 제한 없음.** 학교별로 나누지 않고 위치 위주로 간다

**Supabase 매핑:**

- 이메일: `signInWithOtp` (가입) → `updateUser({ password })` → 이후 `signInWithPassword`
- 카카오: `signInWithOAuth({ provider: 'kakao' })` — Supabase 공식 지원
- 네이버는 채택하지 않는다 (공식 지원 없음, 우회 = 오버엔지니어링)

### 2.2 게스트(비로그인) 접근 범위 (2026-07-19 확정 · 2026-07-21 보강)

**게스트가 볼 수 있는 것은 딱 하나다. 지도에서 마커를 한 번 눌렀을 때 뜨는 카드.**

**카드에 들어가는 것 — 이게 전부:**

- 대표 사진 1장
- 이름
- 최근 밥 언제 먹었는지
- **자유 서술(`description`) 앞 2줄** — 2026-07-22 §2.11로 개정. 원래는 "특이사항(`traits`)"이었다

**성별은 카드에 없다.** 성별은 상세 프로필에만 나온다 (§2.7-1).

**게스트가 할 수 없는 것:**

- 상세 프로필 전체 (다른 사람이 올린 사진들 포함)
- 도감 리스트 · 검색 · 주변 고양이 보기
- 집사 프로필 · 마이캣
- 모든 쓰기 동작 (밥 주기, 목격 등록, 개체 등록)

> ⚠️ **컬럼 차단은 DB 뷰에서 한다** (§2.8-11). 게스트는 `cats_guest`를 읽으므로 성별·중성화는 애초에 내려오지 않는다. 다만 **화면 단위 제한(도감 리스트·검색·집사 프로필 차단)은 여전히 UI 레벨이다.** 컬럼 차단과 화면 차단을 구분해서 읽는다 *(2026-07-21 (4) 정정 — 원래 "전부 화면 레벨"로 적혀 있었다)*

**하단 바는 게스트에게도 4탭 전부 보인다** *(2026-07-21 확정, 구 3-16)*

- 메인홈 외 3탭(주변 고양이 보기·마이캣·집사프로필)은 **숨기지 않고 잠금 상태로 노출**한다. 누르면 화면 전환 없이 로그인 유도를 띄우고 메인홈에 머문다
- 근거 — 앱플로우 §7이 게스트 제한을 "회원가입을 요구하는 요인"으로 규정했다. 숨기면 게스트는 앱에 더 있다는 걸 모른다
- 감수하는 것 — 게스트 체감이 "막힌 앱"이 된다
- 게스트가 잠긴 탭을 눌러도 **해당 화면의 데이터를 미리 가져오지 않는다.** 잠금은 UI이고 실제 차단은 §2.8-11(뷰)·GRANT가 한다

> 구조로 보면 **플로팅 카드 = 게스트 / 상세 페이지 = 로그인 게이트.**

### 2.3 위치 저장 방식 (2026-07-17 확정 · 2026-07-19 재확인)

**write-time 라운딩, 격자 50m.**

```javascript
Math.round(v / 0.00045) * 0.00045
```

- 저장 **직전에** 반올림한다. 정밀 좌표는 DB에 들어간 적이 없다
- 유출돼도 나가는 건 격자까지
- 마커 UX·지도 바탕은 그대로. PostGIS 불필요
- **zones는 사용하지 않는다**
- **노출은 1시간 지연.** 프론트에서 숨기는 게 아니라 **DB가 안 준다.** `cats_guest` / `cats_full` 뷰가 집계 시점에 걸러서 `last_lat` / `last_lng`를 NULL로 내린다 (§2.8-11). **비협상** *(2026-07-21 (4) 정정 — 원래 "RLS"로 적혀 있었으나 실제 구현은 뷰다)*

`sightings` 컬럼:

| 컬럼 | 비고 |
|---|---|
| `lat` / `lng` | 50m 격자로 반올림된 값 |
| `cat_id` | |
| `created_at` | 지연 계산 기준 |
| `deleted_at` | soft-delete |
| `reporter_id` | RLS의 "본인만 등록" 검사에 필요 |
| `photo_path` | 상세 페이지 갤러리에 필요 |

> 랜덤 오프셋이 아닌 이유: 랜덤은 같은 개체 기록이 쌓이면 평균으로 원본이 복원된다. 라운딩은 결정적이라 복원할 잔차가 없다.

> 경도 `0.00045°`는 서울 위도에서 약 40m다. 격자가 50×40m 직사각형이 되는 건 감수한다.

### 2.4 사진 공개 범위 (2026-07-19 확정)

| | 버킷 | 접근 |
|---|---|---|
| **대표 사진 1장** | `cat-covers` (공개) | 누구나. 고정 CDN URL |
| **나머지 사진** | `cat-photos` (비공개) | 로그인 필요. 10분 만료 서명 URL |

- 업로드는 양쪽 다 로그인 필요
- **EXIF 제거를 통과한 파일만 업로드한다** (원칙 4)
- ⚠️ **공개 URL은 회수가 안 된다.** 대표 사진은 "인터넷에 공개한다"는 뜻으로 골라야 한다
- 사진이 1장뿐이면 자동으로 대표사진으로 등록한다

### 2.5 데이터 모델 — 5테이블

`campuses` · `cats` · `sightings` · `feedings` · `profiles`

> **zones는 제외한다.** 좌표가 `sightings`에 들어오면서 거리 계산으로 같은 일이 되므로 할 일이 없어졌다.

- `campuses.status` = `hidden` / `open`
- **`campuses` 테이블과 `cats.campus_id`는 v1에서 미사용이다.** 스키마상 존재하지만 쓰지 않는다. 읽기는 지도 초기 중심에만 남아 있다. 신규 등록 시 `cats.campus_id`는 NULL로 넣는다. `NOT NULL` 제약이 걸려 있으면 nullable로 바꾼다 *(2026-07-21 (3) 확정)*
- 삭제하지 않고 보류하는 이유: Postgres는 컬럼/테이블 추가·삭제 비용이 시점과 무관하므로 지금 지울 실익이 없다. **"나중에 쓰려고 남긴 것"이 아니라 "결정을 유보한 것"이다**
- `feedings`는 `cats`에 FK로 붙는다. 위치 컬럼 없음 (원칙 3). `kind` = 밥/간식/물 (§2.8-15)
- `profiles.role` = `user` / `club` / `admin` — §3-8 참고
- 실물 스키마·정책은 레포의 `docs/db-and-rls.md`에서 관리한다

### 2.6 v1 스코프

**만드는 것 (이것만):**

- 지도 + 마커 · 도감(개체 프로필) · 목격 등록 · 급식 원탭 기록 · 인증(§2.1)
- 주변 고양이 보기 (§2.7-5)

**안 하는 것:**

- **고양이 신규 등록은 로그인한 사용자 누구나** (§2.8-12)

### 2.7 2026-07-21 확정 — §3-1 ~ §3-7 이관

1. **게스트 카드 필드 = 대표사진 · 이름 · 최근 밥 · 자유 서술 2줄.** 성별 제외 *(3-1)* — "특이사항"은 2026-07-22 §2.11에서 자유 서술로 통합됨
2. **"특이사항"과 "건강상태"를 구분하지 않는다.** 건강 관련 내용도 특이사항 한 필드에 *(3-2)*
3. **건강상태 필드는 v1에 만들지 않는다.** `cats`에 별도 건강 컬럼 없음 *(3-3)*
4. **`profiles.campus_id`는 v1에서 채우지 않는다(NULL).** 이 컬럼에 의존하는 로직을 만들지 않는다. 컬럼은 nullable로 유지하고 삭제하지 않는다. 관리자 기능 도입 시 재논의 *(3-4)*
5. **주변 고양이 보기 = v1 포함.** 1시간 지연된 목격만 사용, 길안내·알림·실시간 갱신 금지 *(3-5)*
6. **마이캣 · 집사 프로필 · 급식 카운터 = v1 포함.** 팔로우/좋아요/피드는 v2 *(3-6)*
7. **가입 퀴즈 · 서약서 = v1 포함.** 완료 플래그는 `profiles.onboarded_at` *(3-7)*
8. **사진 0장일 때는 색 기준 기본 제공 일러스트 중 선택.** AI 생성 이미지 아님 *(앱플로우 §8-C)*
9. **카카오 프로필 미저장 유지** *(앱플로우 §8-B)*

### 2.8 2026-07-21 확정 — §3-11 ~ §3-15

**11. 게스트 제한은 DB 뷰로 한다.** (2026-07-21 (4) 정정 — 원래 "화면 레벨"이었음)

anon은 `cats` 테이블을 직접 읽지 않는다. 로그인 여부에 따라 읽는 뷰가 갈린다.

- 게스트 → `cats_guest` (대표사진·이름·특이사항·최근밥·좌표)
- 로그인 → `cats_full` (전체)
- 두 뷰 모두 `sightings` 집계로 `last_lat` / `last_lng`를 만들고, 1시간 지연을 적용한다 (§2.3)
- `cat-photos` 버킷 비공개는 그대로. `photo_path`가 보여도 서명 URL 없이는 못 연다
- 대가 — `cats`에 컬럼이 늘면 뷰 두 개를 같이 고쳐야 한다
- 원래 결정("anon에게 cats를 직접 연다 / 게스트 전용 뷰 만들지 않는다")은 코드를 보지 않고 문서에만 쓴 것이었고, 실제 구현이 이미 뷰 기반이었다. 문서를 코드에 맞춘다
- **검증 완료 (2026-07-21) — 실제로 위반이었고 고쳤다.** `cats_guest`에만 지연이 걸려 있었고 `cats_full`에는 없었다 → 로그인만 하면 실시간 좌표가 나가는 상태였다(원칙 1 위반). 두 뷰 모두에 `sighting_delay()` 필터를 넣어 해결
- **두 뷰는 `security_invoker`가 꺼져 있다** (`reloptions` = null, 2026-07-21 확인). 즉 뷰는 소유자 권한으로 돌며 **기저 테이블의 RLS를 통과한다.** `sightings`의 지연 RLS가 뷰 안에서 안 먹혔던 이유다
- ⚠️ **그래서 게스트 방어선은 RLS가 아니라 GRANT다.** `cats_full`에 `anon` SELECT가 **없어야** 한다 (2026-07-21 확인 — 없음). 뷰를 건드릴 때마다 이걸 다시 확인한다
- **제거됨 — `cats_guest`의 `campuses` INNER JOIN.** `cp.status='open'` 조건이 §2.5·§2.9와 충돌했고, `campus_id`가 NULL인 신규 고양이가 게스트에게만 안 보이는 문제가 있었다

**12. 고양이 신규 등록은 로그인한 사용자 누구나.** `cats` INSERT = `authenticated`.

- "동아리원·관리자만"은 폐기. Phase D와 `CLAUDE.md`에 남아 있던 문구를 지웠다
- role 기반 제한이 필요해지면 Phase F(신고/관리)에서 다시 본다

**13. 카카오 로그인에서 `account_email` 동의항목을 받는다.** 비즈앱 전환 완료(2026-07-20).

- Supabase의 "Allow users without an email"에 의존하지 않는다. 그 옵션을 켜도 카카오 쪽에서 이메일을 요구하는 상황이 확인됐고, 코드로 우회되는 문제가 아니었다
- **검증 필요(코드 아님)** — 같은 주소로 이메일 가입 + 카카오 로그인 시 계정이 1개로 합쳐지는지 2개로 갈리는지. `select id, email from auth.users`로 확인. 2개면 안내 문구가 필요하다

**14. 로그인한 사용자는 모든 `profiles` 행을 읽을 수 있다.** anon은 못 읽는다.

- 상세 페이지의 "누가 밥 줬는지"에 집사 닉네임을 띄우기 위해서다. 못 읽으면 에러 없이 빈칸으로 나온다
- **조건: `profiles`에 민감한 컬럼을 넣지 않는다.** 필요하면 별도 테이블로 분리한다
- 이메일은 `profiles`가 아니라 `auth.users`에 있고, 거긴 열지 않는다

**15. `feedings.kind` = 밥 / 간식 / 물 3종.** 집사 프로필 카운터도 3종.

- 앱플로우 §8-E의 "v1은 밥·간식에 초점"은 폐기한다

### 2.9 2026-07-21 확정 — 조회 기준

**고양이 조회는 캠퍼스 단위가 아니라 좌표 bbox 기준으로 통일한다.**

```sql
WHERE lat BETWEEN ? AND ? AND lng BETWEEN ? AND ?
```

- 캠퍼스 단위 조회(`WHERE campus_id = ?`)는 어디에서도 쓰지 않는다
- bbox는 개념 정의, v1 실제 구현은 전체 조회
- 50m 격자로 반올림된 좌표(§2.3)라도 bbox 쿼리에는 지장이 없다
- 개체 수 수백 규모에서는 성능 차이가 없다. 전체 조회(`select * from cats`)로 두어도 v1에서는 무방하다
- **감수하는 것 — 도감이 "확정된 집합"이 아니게 된다.** "홍대 도감 47마리" 같은 고정 개수가 없고 항상 "지금 지도 범위에 보이는 N마리"다. 원칙 1("우연히 만났을 때 알아보게")상 도감은 컬렉션이 아니라 **식별 도구**이므로 일관된다
- **지도 초기 중심 좌표는 v1에서 `campuses` 테이블에서 읽는다.** 코드 상수화는 보류 *(2026-07-21 (4) 정정)*

### 2.11 2026-07-22 확정 — 고양이 등록 폼·필드

**사용자에게 받는 것 — 이게 전부다.**

| 항목 | 컬럼 | 필수 | 비고 |

**자동 처리 — 사용자가 모른다.**

- `id` `created_at` `status` — DB 기본값
- `created_by` — `auth.uid()`
- `campus_id` — **NULL** (§2.5 · nullable 실물 확인 2026-07-22)
- `code` — **도감번호 자동 생성.** Postgres sequence + trigger. 앱에서 `count(*)+1`을 쓰지 않는다(동시 등록 시 중복). **번호에 큰 의미를 두지 않는다** — 표시용 식별자일 뿐 순서나 총개수를 뜻하지 않는다 (§2.9)

**`traits`는 v1에서 쓰지 않는다.**

- 배열 컬럼이지만 자유 서술 한 칸으로 통합했다. `NOT NULL default '{}'`이라 빈 배열로 남는다. 삭제하지 않고 보류한다(§2.5와 같은 취급)
- 입력과 저장을 어긋내지 않는다 — 자유 서술을 줄바꿈으로 잘라 `traits`에 넣는 방식은 폐기했다

**자유 서술의 역할 — 감상문이 아니라 식별 정보다.**

- 게스트 카드에 **앞 2줄만** 노출된다 (§2.2). 나머지는 상세에서
- 폼 안내 문구로 식별 정보를 유도한다 — 예: *"이 고양이를 알아볼 수 있는 특징을 적어주세요"*
- **감수는 위험** — "귀여운 냥냥이다!!" 같은 감상문으로 채워지면 원칙 1("우연히 만났을 때 알아보게 만든다")이 죽는다. 경쟁 서비스(밥줬냥)가 정확히 그 상태다. **구조로 막을 수 없으므로 카피로 막는다**
- §2.7-2(특이사항과 건강상태를 구분하지 않는다)와 일관된다 — 칸이 하나라 구분할 여지가 없다

> 앱플로우 §3·§7의 "특이사항"은 이 항목의 자유 서술로 대체된다.

**색(`color`) 컬럼 — 신규 추가 확정 (2026-07-22)**

- `cats`에 **색 컬럼이 없었다.** 그런데 세 곳이 색을 전제하고 있다 — 앱플로우 §5(도감 검색 기준), 앱플로우 §4(중복 확인 기준), 로드맵 §2.7-8(사진 0장일 때 색 기준 기본 일러스트)
- 더군다나 §2.11에서 `traits`를 폐기했으므로, 색이 없으면 **도감 검색 조건이 성별 하나만 남는다**
- 등록 폼에서 선택받는다. 나중에 추가하면 폼을 두 번 고치고 기존 개체는 색이 빈다
- **목록은 초안이다 (§3-19).** 치즈 · 고등어 · 삼색 · 카오스 · 턱시도 · 검정 · 하양 · 젖소 — 정우석이 직접 수정할 예정

**등록 확인 단계 (2026-07-22 확정)**

등록 폼보다 먼저 확인 한 단계를 둔다.

- **강제가 아니라 선택이다.** 바로 등록하려는 사람을 막지 않는다
- 도감 화면에 **"새로 등록하기" 버튼**을 둔다. 확인 후 돌아올 경로가 없으면 질문을 또 보게 된다
- **데이터 0건일 때의 분기를 만들지 않는다.** 도감이 비어 있으면 빈 상태 화면(일러스트 + 안내문 한 줄)을 보여주면 된다
- ⚠️ **실제 데이터 수집 전에 Phase B 더미를 지운다.** 안 지우면 첫 등록부터 확인 화면에 더미가 뜬다

**v1에서 만들지 않는 것 (2026-07-22 확정)**

- **위치·색 기준 자동 매칭** — 앱플로우 §4의 "이 고양이가 맞습니까?" 후보 제시. 시스템이 후보를 추리는 건 §2.6이 v2로 미룬 AI 개체인식과 같은 성격이다. **사람이 눈으로 확인하는 단계만 v1에 넣는다**
- **주소검색** — 앱플로우 §4의 "주소검색 / 드래그". 캠퍼스 1개라 지도 드래그로 충분하고, 주소검색은 카카오 로컬 API를 따로 붙여야 해서 스코프 대비 이득이 적다

### 되돌린 것 (같은 실수 반복 방지)

- **좌표를 zone 참조로 저장하려던 코드 → 폐기.** 문서가 "50~100m 퍼징"으로만 적어놔서 재해석 여지가 생겼다
- **대학 이메일 도메인 검증 → 폐기.** 7/18 결정으로 이미 빠진 상태
- **"게스트에게 성별을 보여주는" 프로토타입 UI → 폐기.** §2.2 확정과 어긋난다
- **"고양이 등록은 동아리원·관리자만" → 폐기.** 앱플로우와 로드맵이 정반대로 적혀 있던 항목. §2.8-12로 전체 개방 확정
- **"Allow users without an email로 카카오 이메일 회피" → 폐기.** 실제로는 카카오 쪽에서 이메일을 요구했고 코드로 우회되지 않았다. 비즈앱 전환으로 정면 해결
