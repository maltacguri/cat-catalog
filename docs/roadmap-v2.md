## 0. 이 문서를 쓰는 법

**이 문서가 유일한 기준이다.** 채팅 기록, 앱플로우 이미지, 예전 로드맵보다 이 문서가 우선한다.

지금까지 같은 결정을 여러 번 다시 논의한 이유는 확정 사항이 여러 채팅에 흩어져 있었고, 문서가 낡은 채로 남아 있었기 때문이다. 그래서 규칙 세 개:

1. **§2에 있는 건 다시 논의하지 않는다.** 바꾸려면 논의가 아니라 **이 문서를 고치고 §5에 이력을 남긴다.**
2. **§3에 없는 건 미확정이 아니다.** 새 아이디어는 §3에 추가한 뒤에 논의한다.
3. **확정과 미확정을 한 문장에 섞지 않는다.** 애매하면 미확정이다.

## 1. 정의와 원칙

### 한 줄 정의

한 대학 캠퍼스의 길고양이를 지도 기반으로 아카이빙하는 서비스.
"저 고양이 이름 뭐지?"라는 호기심으로 들어와서, "3시간 전 밥 먹음"을 보고 돌봄으로 이어지게 만든다.

### 차별점

급식 로그를 **개인 관리 기록이 아니라 낯선 사람에게 보내는 공개 신호**로 다룬다.
그리고 고정 급식소를 만들지 않는다.

### 원칙 5개 (기능 판단 기준)

**1. "찾아가게" 만들지 말고 "우연히 만났을 때 알아보게" 만든다.**

- **금지** — 실시간 위치, 위치 기반 푸시/알림, 경로 안내, 실시간 거리 갱신, 특정 개체 추적
- **허용** — 1시간 지연된 목격 기록을 조회·정렬·필터하는 정적 화면
- `주변 고양이 보기`(§2.7-5)는 허용 쪽이다. **지연을 우회하는 순간 금지 쪽으로 넘어간다**

> 2026-07-21 문구 수정. 원래는 "실시간 위치 알림·경로 안내 없음"이었는데, §3-5(주변 고양이 보기)를 v1에 넣기로 하면서 원칙과 정면 충돌했다. 금지 대상은 **"지금 여기 있다"**이지 **"이 근처에 이런 애들이 산다"**가 아니라는 선으로 좁혔다.

**2. 급식소는 만들지 않는다.** 고정 급식 장소 등록 기능 없음.

**3. 급식 기록은 장소가 아니라 고양이 개체에 붙는다.**

**4. 사진의 EXIF(GPS 포함)는 무조건 제거한다.**

**5. 목격 좌표는 저장 시점에 50m 격자로 반올림한다.** 정밀 좌표는 DB에 존재하지 않는다.

> ⚠️ 원칙 5는 v1에서 "50~100m 퍼징"이라고만 적혀 있었다. 그 문장이 모호해서 구현이 두 번 뒤집혔다. 이제 방식까지 못 박는다. **랜덤 오프셋이 아니라 결정적 라운딩이다.**

## 2. 확정 사항 — 다시 논의하지 않는다

### 2.1 인증 · 회원가입 (2026-07-19 확정)

**두 갈래를 모두 제공한다.**

| 방식 | 회원가입 | 이후 로그인 |
|---|---|---|
| **이메일** | 이메일 OTP 인증 **1회** → 비밀번호 설정 | 이메일 + 비밀번호 |
| **카카오** | 카카오 인증 (`account_email` 동의항목 포함, §2.8-13) | 카카오 인증 (계속) |

**두 방식 공통:**
- 가입 시 앱 자체 프로필을 새로 만든다
- **집사 이름(닉네임)은 무작위로 자동 생성**한다
- 카카오의 닉네임·프로필 사진은 **저장하지도, 화면에 그리지도 않는다.** 카카오는 오직 인증 통과용
- **대학 이메일 도메인 제한 없음.** 학교 메일이 없는 사람이 있고, 학교 메일 서버가 외부 인증 메일을 막는 사고가 잦다. 학교별로 나누지 않고 위치 위주로 간다

**Supabase 매핑:**
- 이메일: `signInWithOtp` (가입) → `updateUser({ password })` → 이후 `signInWithPassword`
- 카카오: `signInWithOAuth({ provider: 'kakao' })` — Supabase 공식 지원
- 네이버는 채택하지 않는다 (공식 지원 없음, 우회 = 오버엔지니어링)

---

### 2.2 게스트(비로그인) 접근 범위 (2026-07-19 확정 · 2026-07-21 보강)

**게스트가 볼 수 있는 것은 딱 하나다. 지도에서 마커를 한 번 눌렀을 때 뜨는 카드.**

**카드에 들어가는 것 — 이게 전부:**
- 대표 사진 1장
- 이름
- 최근 밥 언제 먹었는지
- 특이사항

**성별은 카드에 없다.** 성별은 상세 프로필에만 나온다. (§3-1, 2026-07-21)

**게스트가 할 수 없는 것:**
- 상세 프로필 전체 (다른 사람이 올린 사진들 포함)
- 도감 리스트 · 검색 · 주변 고양이 보기
- 집사 프로필 · 마이캣
- 모든 쓰기 동작 (밥 주기, 목격 등록, 개체 등록)

> ⚠️ **위 목록은 화면 레벨 제한이다** (§2.8-11). anon 키로 `cats`를 직접 조회하면 성별·중성화·전체 목록이 나온다. 데이터 차단이 아니라 UI 차단이라는 걸 알고 쓴다.
> 구조로 보면 **플로팅 카드 = 게스트 / 상세 페이지 = 로그인 게이트.**

---

### 2.3 위치 저장 방식 (2026-07-17 확정 · 2026-07-19 재확인)

**write-time 라운딩, 격자 50m.**

```js
Math.round(v / 0.00045) * 0.00045
```

- 저장 **직전에** 반올림한다. 정밀 좌표는 DB에 들어간 적이 없다
- 유출돼도 나가는 건 격자까지
- 마커 UX·지도 바탕은 그대로. PostGIS 불필요
- **zones는 사용하지 않는다** (§5 변경 이력 참고)
- **노출은 1시간 지연.** 프론트에서 숨기는 게 아니라 **RLS가 안 준다.** §2.8-11(화면 레벨 제한)의 예외다 — 지연만은 데이터 레벨이다

`sightings` 컬럼:

| 컬럼 | 비고 |
|---|---|
| `lat` / `lng` | 50m 격자로 반올림된 값 |
| `cat_id` | |
| `created_at` | 지연 계산 기준 |
| `deleted_at` | soft-delete |
| `reporter_id` | 구현상 추가 — RLS의 "본인만 등록" 검사에 필요 |
| `photo_path` | 구현상 추가 — 상세 페이지 갤러리에 필요 |

> 랜덤 오프셋이 아닌 이유: 랜덤은 같은 개체 기록이 쌓이면 평균으로 원본이 복원된다. 라운딩은 결정적이라 복원할 잔차가 없다.
> 경도 `0.00045°`는 서울 위도에서 약 40m다. 격자가 50×40m 직사각형이 되는 건 감수한다.

### 2.4 사진 공개 범위 (2026-07-19 확정)

| | 버킷 | 접근 |
|---|---|---|
| **대표 사진 1장** | `cat-covers` (공개) | 누구나. 고정 CDN URL |
| **나머지 사진** | `cat-photos` (비공개) | 로그인 필요. 10분 만료 서명 URL |

- 업로드는 양쪽 다 로그인 필요
- **EXIF 제거를 통과한 파일만 업로드한다** (원칙 4)
- ⚠️ **공개 URL은 회수가 안 된다.** 파일을 지워도 이미 퍼진 링크·복사본은 못 막는다. 대표 사진은 "인터넷에 공개한다"는 뜻으로 골라야 한다
- 사진이 1장뿐이면 자동으로 대표사진으로 등록한다 (2026-07-19)

### 2.5 데이터 모델 — 5테이블

`campuses` · `cats` · `sightings` · `feedings` · `profiles`

> **zones는 제외한다.** 원래 용도는 "도감에 80마리면 목록 스크롤을 아무도 안 한다"를 풀기 위해 구역으로 후보를 좁히는 것(`cats.zone_id`)이었다. 좌표가 `sightings`에 들어오면서 거리 계산으로 같은 일이 되므로 할 일이 없어졌다.

- `campuses.status` = `hidden` / `open`. 데이터가 부족한 캠퍼스는 앱에서 안 보인다
- `feedings`는 `cats`에 FK로 붙는다. **위치 컬럼 없음** (원칙 3). `kind` = 밥/간식/물 (§2.8-15)
- `profiles.role` = `user` / `club` / `admin` — §3-8 참고
- 실물 스키마·정책은 `docs/db-and-rls.md`에서 관리한다

### 2.6 v1 스코프

**만드는 것 (이것만):**
- 지도 + 마커
- 도감 (개체 프로필)
- 목격 등록
- 급식 원탭 기록
- 인증 (§2.1)
- 주변 고양이 보기 (§2.7-5)
- 마이캣(북마크) · 집사 프로필 · 급식 카운터 (§2.7-6)
- 가입 퀴즈 · 서약서 (§2.7-7)

**안 하는 것 (나중, 지금 손대면 스코프 터짐):**
- 커뮤니티 / 게시판
- AI 고양이 자동 인식
- 실종 매칭
- 팔로우 / 좋아요 / 피드

**확정된 권한 규칙:**

- 과급식 방지: 최근 급식 기록이 너무 최근이면 한 번 되묻는다
- **고양이 신규 등록은 로그인한 사용자 누구나** (§2.8-12)

### 2.7 2026-07-21 확정 — §3-1 ~ §3-7 이관

1. **게스트 카드 필드 = 대표사진 · 이름 · 최근 밥 · 특이사항.** 성별 제외. 성별은 상세 프로필에만 *(3-1)*
2. **"특이사항"과 "건강상태"를 구분하지 않는다.** 건강 관련 내용도 특이사항 한 필드에 넣는다 *(3-2)*
3. **건강상태 필드는 v1에 만들지 않는다.** `cats`에 별도 건강 컬럼 없음 *(3-3)*
4. **`profiles.campus_id`는 v1에서 채우지 않는다(NULL).** 컬럼은 남기되 이 컬럼에 의존하는 로직을 만들지 않는다 *(3-4)*
5. **주변 고양이 보기 = v1 포함.** 조건 — 1시간 지연된 목격만 사용, 길안내·알림·실시간 갱신 금지, 검색축은 색·성별·특이사항 *(3-5)*
6. **마이캣(북마크) · 집사 프로필 · 급식 카운터 = v1 포함.** 팔로우/좋아요/피드는 v2 *(3-6)*
7. **가입 퀴즈 · 서약서 = v1 포함.** 완료 플래그는 `profiles.onboarded_at` *(3-7)*
8. **사진 0장일 때는 색 기준 기본 제공 일러스트 중 선택.** AI 생성 이미지 아님 *(앱플로우 §8-C)*
9. **카카오 프로필 미저장 유지** — 앱플로우 §8-B는 §2.1 쪽으로 정리됨

---

### 2.8 2026-07-21 확정 — §3-11 ~ §3-15

**11. 게스트 제한은 화면 레벨로 한다.** anon에게 `cats` SELECT를 직접 연다. 게스트 전용 뷰/RPC는 만들지 않는다.
- 감수하는 것 — 성별 · 중성화 여부 · 전체 목록이 API로 조회 가능하다
- **감수하지 않는 것 — 1시간 지연.** 이건 그대로 RLS다 (원칙 1 · §2.3, 비협상)
- `cat-photos` 버킷 비공개도 그대로다. `photo_path` 문자열이 보여도 서명 URL 없이는 못 연다
- 되돌리려면 뷰를 추가하고 anon 정책 + 프론트 쿼리 대상만 바꾸면 된다. 못 박히는 결정이 아니다

**12. 고양이 신규 등록은 로그인한 사용자 누구나.** `cats` INSERT = `authenticated`.
- "동아리원·관리자만"은 폐기. 로드맵 Phase D와 CLAUDE.md에 남아 있던 문구를 지웠다
- role 기반 제한이 필요해지면 Phase F(신고/관리)에서 다시 본다

**13. 카카오 로그인에서 `account_email` 동의항목을 받는다.** 비즈앱 전환 완료(2026-07-20).
- Supabase의 "Allow users without an email"에 의존하지 않는다. 그 옵션을 켜도 카카오 쪽에서 이메일을 요구하는 상황이 확인됐고, 코드로 우회되는 문제가 아니었다
- **검증 필요(코드 아님)** — 같은 주소로 이메일 가입 + 카카오 로그인 시 계정이 1개로 합쳐지는지 2개로 갈리는지. `select id, email from auth.users`로 확인. 2개면 안내 문구가 필요하다

**14. 로그인한 사용자는 모든 `profiles` 행을 읽을 수 있다.** anon은 못 읽는다.
- 상세 페이지의 "누가 밥 줬는지"에 집사 닉네임을 띄우기 위해서다. 못 읽으면 에러 없이 빈칸으로 나온다
- **조건: `profiles`에 민감한 컬럼을 넣지 않는다.** 필요하면 별도 테이블로 분리한다
- 이메일은 `profiles`가 아니라 `auth.users`에 있고, 거긴 열지 않는다

**15. `feedings.kind` = 밥 / 간식 / 물 3종.** 집사 프로필 카운터도 3종.
- 앱플로우 §8-E의 "v1은 밥·간식에 초점"은 폐기한다

## 3. 미확정 — 여기서만 논의한다

> 확정하면 §2로 옮기고 §5에 이력을 남긴다.
> **2026-07-21: 3-1 ~ 3-7은 §2.7로, 3-11 ~ 3-15는 §2.8로 이관 완료.**
> **지금 열려 있는 건 아래 3개뿐이고, 셋 다 Phase B~C를 막지 않는다.**

**3-8. `role` 모델 확정** — `user` / `club` / `admin` 3단으로 갈지, 캠퍼스별 권한을 어떻게 묶을지.
§3-12가 "누구나 등록"으로 닫히면서 급하지 않아졌다. Phase F(신고/관리) 착수 전까지 정하면 된다.

**3-10. "배포"의 정의** — 소프트런치(신뢰 그룹) / 캠퍼스 전체 공개 중 어디가 목표인지.

### 되돌린 것 (같은 실수 반복 방지)

- **좌표를 zone 참조로 저장하려던 코드 → 폐기.** 7/17에 확정된 50m 라운딩을 문서가 "50~100m 퍼징"으로만 적어놔서 재해석 여지가 생겼다. 원칙 5의 문구를 방식까지 명시하도록 고쳤다(§1)
- **대학 이메일 도메인 검증 → 폐기.** 로드맵 v1에 남아 있던 항목인데 7/18 결정으로 이미 빠진 상태
- **"게스트에게 성별을 보여주는" 프로토타입 UI → 폐기.** §2.2 확정과 어긋난다
- **"고양이 등록은 동아리원·관리자만" → 폐기.** 앱플로우와 로드맵이 정반대로 적혀 있던 항목. §2.8-12로 전체 개방 확정
- **"Allow users without an email로 카카오 이메일 회피" → 폐기.** 실제로는 카카오 쪽에서 이메일을 요구했고 코드로 우회되지 않았다. 비즈앱 전환으로 정면 해결
